## Component Inventory

The page has exactly four rendered elements. No component library is used; each is a plain HTML element with Tailwind classes.

| Element | Tag / Pattern | Notes |
|---------|--------------|-------|
| Page title | `<h1>` | "Selfie Test Tester" — static, always visible |
| Verify button | `<button>` | Triggers `startWidgetFlow()` on click; disabled while widget is open |
| Status text | `<p>` | Conditionally rendered; three states below |
| IDKitRequestWidget | Library component | Renders as modal overlay; library owns all QR UI |

### State Machine

The page has three mutually exclusive states:

| State | Trigger | Status text shown |
|-------|---------|-------------------|
| `idle` | Initial load, or after reset | Nothing (status area hidden) |
| `verified` | `onSuccess` callback from IDKitRequestWidget | "Verified!" in green-600 |
| `error` | `onError` callback from IDKitRequestWidget | Error message + code in red-600, monospace |

Button text is always "Start Verification". Button is `disabled` (opacity-50, cursor-not-allowed) while the widget modal is open (i.e., between button click and widget close/complete).

## Interaction Contract

| Interaction | Trigger | Response |
|-------------|---------|----------|
| Click "Start Verification" | User clicks button | Calls `startWidgetFlow()`, button becomes disabled |
| Widget opens | `IDKitRequestWidget open={true}` | Library renders modal overlay with QR |
| Widget closes (user dismisses) | User closes modal without completing | Button re-enabled, status unchanged |
| Verification succeeds | `onSuccess` fires | Modal closes, status switches to `verified` state |
| Verification fails | `onError` fires | Modal closes, status switches to `error` state with error code |

No loading spinner required — the widget modal itself provides visual feedback during the QR scan flow. The only loading gap is the instant between button click and modal open, which is imperceptible (no async fetch needed; `rp_context` is fetched inside the widget flow).

Wait — per STATE.md decision: "rp_context fetched on button click, not page load (prevents expiry before scan)." This means there IS a brief async gap on button click before the widget can open. Button must show a loading state during this fetch.

| Loading interaction | Trigger | Response |
|--------------------|---------|----------|
| rp_context fetch pending | Button clicked | Button disabled, text changes to "Loading..." |
| rp_context fetch complete | API responds | `startWidgetFlow()` called, button text reverts to "Start Verification" (still disabled while modal open) |
| rp_context fetch fails | API error | Button re-enabled, status shows error state with message |
